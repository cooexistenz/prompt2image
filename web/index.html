How It Works

Three steps from text to a token-priced image, with the math shown at every step.

1

Send Your Prompt

POST /render with JSON. Pick a profile — claude, openai, gemini, or ocr — each tuned to how that consumer resamples and bills images.

2

Get Pages + the Honest Math

You receive PNG pages and a token report: estimated text tokens vs. image tokens per provider. When plain text is cheaper, the report says so instead of hiding it.

3

Attach Anywhere

Feed the pages to any vision model or OCR engine. Line breaks survive as visible ↵ marks; the exact original can also ride along as PNG metadata.

Built for Real Workflows

Provider-Tuned Profiles

Page geometry snaps to each provider's billing grid: Claude's 1568 px edge, OpenAI's 512 px tiles, Gemini's 768 px tiles. No wasted pixels, no surprise resamples.

Honest Token Report

Every render returns text-token and image-token estimates side by side. The tool recommends against itself when text wins — that's the point.

Classic OCR Mode

The ocr profile renders 16×32 px glyphs with real line breaks and generous margins — tuned for Tesseract, EasyOCR, and PaddleOCR rather than token count.

Privacy by Architecture

Not a policy — a design. Prompts render entirely in memory and are never written to disk or logged. Access logs carry byte counts and timings only.

Lossless Round-Trip

Hard newlines become visible ↵ marks, tabs become → marks, and embedOriginal stores the exact prompt as PNG metadata for byte-perfect recovery.

Zero Dependencies

Pure Node 18+. The PNG encoder, bitmap font atlas, and REST server are all in-tree — npm install pulls nothing at runtime.

The Honest Part

Rendering text as pixels is a trade, not magic. Prompt2Image is built to make the trade visible:

  • Vision models don't run OCR. They read images through patch embeddings — dense renders work remarkably well on the strongest current models, but exact strings (IDs, hashes, long numbers) can be silently misread. Keep byte-critical values in plain text.
  • Sometimes text is simply cheaper. Fixed tile costs mean small prompts lose money as images on some providers. The report flags this per render instead of pretending otherwise.
  • Readability differs by model. Test the dense profile against your target model before relying on it; use the ocr profile when a classic OCR engine is the reader.

A One-Endpoint API

Stateless by design. Send JSON, get a PNG or base64 pages — nothing is persisted between requests.

# Single-page PNG straight to a file
curl -s http://localhost:8018/render \
  -H "content-type: application/json" \
  -H "accept: image/png" \
  -d '{"prompt": "Your prompt here", "profile": "claude"}' \
  -o prompt.png

# JSON mode: all pages as base64 + the full token report
curl -s http://localhost:8018/render \
  -H "content-type: application/json" \
  -d '{"prompt": "Your prompt here", "profile": "gemini",
       "options": {"embedOriginal": true}}' | jq .report

# Self-host in under a minute
git clone https://github.com/cooexistenz/prompt2image
cd prompt2image && npm install && npm run build && npm start

Questions, Answered Plainly

Is an image always cheaper than text?

No — and the tool never claims it is. Dense rendering wins on pixel-billed providers (Claude bills ≈ pixels ÷ 750) once the prompt is long enough, and loses on tile-billed providers for short prompts. Every response includes both estimates so you decide with numbers, not vibes.

Why is the background white instead of transparent?

Transparency is flattened unpredictably downstream — many pipelines composite alpha onto black, making dark text invisible, and OCR engines binarize badly against alpha. White is the reliable default; pass "background": "transparent" when you genuinely need overlay output.

Can the exact original prompt be recovered from the image?

Two ways. Visually: hard newlines are drawn as ↵ marks and tabs as → marks, so the layout reconstructs. Byte-perfect: set embedOriginal and the exact string ships inside the PNG as an iTXt metadata chunk.

Is my prompt stored anywhere?

No. Rendering happens in memory, the PNG is returned, and the buffers are garbage-collected. There is no database, no cache, no prompt logging — and it's open source, so you can verify that instead of trusting it.

Which characters are supported?

Full printable ASCII plus Western European Latin (umlauts, ß, accents, €) in both modes; the OCR atlas covers over 1,000 glyphs. Unsupported characters are counted and listed in the report as droppedChars — never silently mangled.
